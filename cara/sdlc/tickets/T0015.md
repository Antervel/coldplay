## T0015 — Fix resource cleanup in `terminate/2` and orphaned process handling
git branch: `ai-agent/t0015_fix_terminate_and_cleanup`

**Problem:**
Per RULE 40, resources (monitors, timers, ETS tables, subscriptions, spawned processes) should be released on every exit path. Two issues exist:

**Issue 1 — `ChatLive.terminate/2` crashes when `chat_id` is missing** (`lib/cara_web/live/chat_live.ex:62-65`):

```elixir
def terminate(_reason, socket) do
  Session.broadcast_left(socket, socket.assigns.chat_id)
  :ok
end
```

If `mount/3` redirects before `chat_id` is assigned (e.g., student info not provided → redirect to `/student`), then `socket.assigns.chat_id` doesn't exist. Accessing it in `terminate/2` raises a `KeyError`, which causes the terminate callback itself to crash. Phoenix handles this gracefully (the crash is logged but the process still exits), but the crash noise is confusing and the broadcast never happens.

The `Session.broadcast_left/2` function at `session.ex:126` already has a guard:

```elixir
def broadcast_left(socket, chat_id) do
  if Map.has_key?(socket.assigns, :chat_id) and Monitoring.monitoring_enabled?() do
    Monitoring.broadcast_chat_left(socket, chat_id)
  end
  :ok
end
```

But the `terminate/2` still accesses `socket.assigns.chat_id` directly on line 63, before `broadcast_left` can check. The fix is to guard `chat_id` access at the `terminate` level.

**Issue 2 — Orphaned LLM processes when LiveView crashes** (`lib/cara_web/live/chat_live.ex:378`):

```elixir
{:ok, pid} = ChatOrchestrator.run(llm_call_params)
branched_chat = BranchedChat.set_active_task(branched_chat, branch_id, pid, message)
```

When a LiveView process is running an LLM call and crashes (for any reason), the spawned `ChatOrchestrator` process keeps running. It becomes orphaned — nobody is listening to its stream events anymore. It will eventually finish its work and try to send messages to a dead process, failing silently. This wastes resources.

**Issue 3 — `MetadataHandle.start_link` leak on error paths** (`lib/cara/ai/chat.ex:200-201`):

```elixir
defp dummy_stream_response(%StreamResponse{context: context, model: model}) do
  {:ok, metadata_handle} = MetadataHandle.start_link(fn -> %{} end)
  ...
```

`MetadataHandle.start_link` spawns a linked process. In normal flow, the caller consumes the stream response and the metadata process exits naturally. But on error paths (e.g., `handle_stream_for_tools` raises at line 189-190), the `dummy_stream_response/1` is called but the metadata process is never properly cleaned up.

**Solution:**

**Fix 1:** Guard the `chat_id` access in `terminate/2`:

```elixir
def terminate(_reason, socket) do
  if Map.has_key?(socket.assigns, :chat_id) do
    Session.broadcast_left(socket, socket.assigns.chat_id)
  end
  :ok
end
```

**Fix 2:** Monitor orphaned LLM processes and clean them up in `terminate/2`:

```elixir
def terminate(_reason, socket) do
  if Map.has_key?(socket.assigns, :chat_id) do
    Session.broadcast_left(socket, socket.assigns.chat_id)
  end

  # Kill any active LLM tasks to prevent orphaned processes
  if Map.has_key?(socket.assigns, :branched_chat) do
    socket.assigns.branched_chat.branches
    |> Enum.each(fn {_branch_id, branch} ->
      if branch.active_task do
        Process.exit(branch.active_task, :kill)
      end
    end)
  end

  :ok
end
```

**Fix 3:** Add a `Process.monitor` on the ChatOrchestrator PID and handle the DOWN message to trigger cleanup. Or, use `Task.Supervisor` which provides this automatically (see T0013).
