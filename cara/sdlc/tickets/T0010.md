## T0010 — Add authentication and tenant-scoped access to audit logs
git branch: `ai-agent/t0010_auth_and_idor_protection`

**Problem:**
Per RULE 14 and RULE 15, the application has **no authentication** and the audit log routes are completely open — anyone who guesses a `chat_id` can read any student's conversation.

**Violations:**

1. **No authentication at all** — Every route in `lib/cara_web/router.ex` is in the public `:browser` pipeline. There is no login, no session, no user concept.

2. **IDOR on logs routes** (`lib/cara_web/controllers/logs_controller.ex:33-49`):
```elixir
def show(conn, %{"chat_id" => chat_id, "branch_id" => branch_id}) do
  session = Audit.get_session(chat_id)
  messages = Audit.list_messages_for_branch(chat_id, branch_id)
```
Anyone who visits `/logs/some-chat-id/some-branch-id` can see all messages for that branch. The `chat_id` and `branch_id` are UUID-like strings but not secret — they could be guessed, leaked, or discovered from other parts of the UI.

3. **Unscoped data access in `Cara.Audit`** (`lib/cara/audit.ex`):
   - `get_session/1` at line 31 — returns any session by `chat_id`, no ownership check
   - `list_branches/1` at line 58 — lists **all** sessions/branches globally
   - `list_messages_for_branch/2` at line 130 — returns messages with no user/tenant filter

**Solution:**

Implement a simple shared-secret authentication scheme (or basic auth) for the logs routes. The full scope of this ticket:

**Step 1: Add basic auth to all routes** (or a shared secret header for the logs section)

The simplest approach that prevents IDOR and unauthorized access:
- Add a shared secret (environment variable `LOGS_AUTH_TOKEN`) to the app config
- Add a new plug pipeline `:logs_auth` that checks for a matching token in the session or as a URL query param
- Apply the pipeline to the logs routes

**Step 2: Scope `Cara.Audit` functions to a tenant/user**

Since the app doesn't have a multi-user system yet, the simplest approach is to add authentication that gates access to the logs routes entirely. For future-proofing, the `Audit` context functions should accept an optional scope parameter.

**Alternative simpler approach: Add basic HTTP auth**

Use Phoenix's `Plug.BasicAuth` to protect the logs routes with a username/password:

```elixir
# In router.ex
pipeline :logs_auth do
  plug Plug.BasicAuth, use_config: {:cara, :logs_auth}
end

scope "/", CaraWeb do
  pipe_through [:browser, :logs_auth]
  
  get "/logs", LogsController, :index
  get "/logs/:chat_id/:branch_id", LogsController, :show
end
```

In config:
```elixir
config :cara, :logs_auth,
  username: "admin",
  password: {:system, "LOGS_PASSWORD"}
```

**Step 3 (future): Create a user system** — out of scope for this ticket. This ticket covers the minimum viable auth to prevent unauthorized access.
