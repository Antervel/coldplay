## T0007 — Migrate `ChatLive` messages to LiveView streams
git branch: `ai-agent/t0007_chat_live_streams

**Problem:**
Per DMV Prompt Rule 2, collections that can exceed ~100 items should use LiveView streams to avoid O(n) memory per connected user. `ChatLive` stores all messages inside a `BranchedChat` struct assigned via `assign(socket, :branched_chat, branched_chat)`. As conversations grow (long tutoring sessions can generate hundreds of messages), the entire struct is sent to the client on every update.

With branching (multiple conversations in one session), the total message count multiplies. `PHX_SERVER=true` deployments are especially sensitive since every connected user keeps the full message list in server memory.

**Solution:**
Replace the current message rendering approach (which reads from `BranchedChat.get_current_messages/1` in the ViewModel) with LiveView streams. Each message in the current branch becomes a stream item. When switching branches, the stream is reset with the new branch's messages.

Key design decisions:
- Stream key is `message.id` combined with `branch_id` to maintain stability across branch switches
- `stream/3` with `reset: true` when switching branches (replaces old message set)
- `stream_insert/4` for new messages (both user and assistant)
- `stream_delete/3` for deleted messages
- A separate `@message_count` assign tracks count (streams don't support `length/1`)
- Streaming chunks (`handle_info({:llm_chunk, ...})`) still update the in-memory `BranchedChat` and use `stream_insert` for the streaming message
- The `ViewModel` module needs updating to not derive messages from `BranchedChat` but from the stream

### Risks and mitigations

- **Risk**: Streaming chunks update the message content rapidly. `stream_insert` on every chunk could cause excessive diffing.
  **Mitigation**: Batch chunk updates — only `stream_insert` every N chunks or on a debounce timer for active streaming messages.

- **Risk**: Branch switching with `reset: true` causes a full re-render of all messages, which could be slow for very long conversations.
  **Mitigation**: Acceptable for an education tool (conversations rarely exceed 500 messages). If it becomes a problem, paginate with `stream` + offset.
