## T0017 — Replace unsafe `hd()` call on external API response in `Wikipedia`
git branch: `ai-agent/t0017_safe_wikipedia_parse_full_article`

**Problem:**
Per RULE 34, bang functions like `hd/1` are appropriate only when failure indicates a programmer bug, not when operating on external data. The `parse_full_article_response` function in `lib/cara/wikipedia.ex:199-206` uses `hd()` on data returned from the Wikipedia API, which is external and unpredictable:

```elixir
defp parse_full_article_response(summary, content) do
  plain_text_content =
    content
    |> Map.get("query")
    |> Map.get("pages")
    |> hd()                          # <-- crashes if pages is empty or missing
    |> Map.get("extract")
```

If the Wikipedia API returns an empty `pages` list (article deleted, page not found, API change, rate limiting with degraded response), `hd/1` raises `ArgumentError`. This crashes whichever process called it — typically an AI tool execution within the LLM chat.

**Context:** The `Wikipedia` module at `lib/cara/wikipedia.ex` is used by `Cara.AI.Tools.Wikipedia` as a tool for the LLM. When the LLM decides to look up a Wikipedia article, this function parses the response. A crash here means the LLM tool call fails abruptly instead of gracefully returning an error message.

**Solution:**

Replace the pipeline with `case` that handles empty/missing pages gracefully:

```elixir
defp parse_full_article_response(summary, content) do
  plain_text_content =
    content
    |> Map.get("query")
    |> Map.get("pages")
    |> case do
      [page | _] when is_map(page) -> Map.get(page, "extract", "")
      _ -> ""
    end

  %{
    title: summary["title"],
    extract: summary["extract"],
    content: plain_text_content,
    url: summary["content_urls"]["desktop"]["page"],
    image: summary["originalimage"]["source"]
  }
end
```

The pattern `[page | _]` destructures the list safely — if the list is empty, it falls through to `_ -> ""`. This converts a crash into a graceful empty-content response.

Alternatively, return an error tuple:
```elixir
defp parse_full_article_response(_summary, content) do
  case content |> Map.get("query") |> Map.get("pages") do
    [page | _] when is_map(page) ->
      %{
        title: content["title"],
        extract: content["extract"] || Map.get(page, "extract", ""),
        content: Map.get(page, "extract", ""),
        url: summary["content_urls"]["desktop"]["page"],
        image: summary["originalimage"]["source"]
      }
    _ ->
      %{title: "Not found", extract: "", content: "", url: "", image: ""}
  end
end
```

**Important:** Notice that `summary` is used for some fields (`title`, `extract`, `url`, `image`) while `content` provides the full text. The `summary` argument is already validated by the time this function is called (it goes through `parse_article_response` first), so the risk is primarily in the `content` path.
