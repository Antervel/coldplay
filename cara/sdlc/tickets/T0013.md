## T0013 — Use supervised `Task.Supervisor` instead of unsupervised `Task.start` for LLM calls
git branch: `ai-agent/t0013_supervised_llm_tasks`

**Problem:**
Per RULE 17, any long-lived process needs to sit under a supervisor — otherwise a crash silently disappears instead of restarting or being observable.

The `ChatOrchestrator.run/1` function in the `branched_llm` dependency uses `Task.start/1` to run LLM calls:

```elixir
# deps/branched_llm/lib/branched_llm/chat_orchestrator.ex:55-57
def run(params) do
  Task.start(fn -> do_process(params) end)
end
```

`Task.start/1` is **unsupervised** — if the spawned process crashes during an LLM call (network timeout, API error, etc.), the crash is silently lost. The caller never gets an error signal.

This is called from `lib/cara_web/live/chat_live.ex:378`:

```elixir
{:ok, pid} = ChatOrchestrator.run(llm_call_params)
```

The app **already has** a `Task.Supervisor` registered in `application.ex:17`:

```elixir
{Task.Supervisor, name: Cara.TaskSupervisor}
```

But `ChatOrchestrator` (in deps) doesn't use it.

**Solution:**

The `ChatOrchestrator` is in the `branched_llm` dependency, so we can't modify it directly. Instead, replace the call to `ChatOrchestrator.run/1` with an equivalent call using `Task.Supervisor.start_child/2` in `chat_live.ex`.

**Step 1:** Create a wrapper in `Cara.AI.Chat` (or a new module) that starts the LLM processing under the app's `TaskSupervisor`:

```elixir
# In lib/cara/ai/chat.ex or a new wrapper module
def start_llm_task(llm_call_params) do
  Task.Supervisor.start_child(Cara.TaskSupervisor, fn ->
    # This replicates what ChatOrchestrator.run/1 does internally
    ChatOrchestrator.do_process(llm_call_params)
  end)
end
```

Wait — `ChatOrchestrator.do_process/1` is a private function. We need to check if there's a way to use the orchestrator with a supervisor.

**Alternative approach:** Instead of using `ChatOrchestrator.run/1`, start a supervised Task that calls `ChatOrchestrator.run/1` inside it:

```elixir
# In chat_live.ex
@impl true
def handle_info(:start_llm, socket) do
  llm_call_params = build_llm_call_params(socket)
  
  {:ok, pid} = Task.Supervisor.start_child(Cara.TaskSupervisor, fn ->
    # ChatOrchestrator.run starts its own Task internally,
    # but we wrap it so the supervisor monitors the outer task
    {:ok, pid} = ChatOrchestrator.run(llm_call_params)
    # Monitor the inner task so we know if it crashes
    ref = Process.monitor(pid)
    receive do
      {:DOWN, ^ref, :process, ^pid, reason} ->
        send(self(), {:llm_crashed, reason})
    end
  end)
  
  branched_chat = BranchedChat.set_active_task(socket.assigns.branched_chat, branch_id, pid, message)
  {:noreply, assign_branched_chat(socket, branched_chat)}
end
```

Actually, looking more carefully, the issue is more nuanced. `ChatOrchestrator.run/1` uses `Task.start/1` internally and returns the PID. The caller stores this PID and later kills it via `Process.exit(pid, :kill)`. The core problem is that if the inner Task crashes, nobody knows.

**Simplest fix:** In `chat_live.ex`, replace the direct call to `ChatOrchestrator.run/1` (line 378) with a supervised version:

```elixir
# Before (line 378):
{:ok, pid} = ChatOrchestrator.run(llm_call_params)

# After:
{:ok, pid} = Task.Supervisor.start_child(Cara.TaskSupervisor, fn ->
  # ChatOrchestrator.run internally does Task.start, but we can
  # run the orchestrator logic directly here if the function is accessible
  ChatOrchestrator.run(llm_call_params)
  # Keep the process alive until the parent kills it or it finishes
  :timer.sleep(:infinity)
end)
```

Hmm, this approach doesn't cleanly work because `ChatOrchestrator.run/1` already returns a PID from an internal `Task.start`.

**Better approach:** Check if `ChatOrchestrator` exposes the internal processing function. If not, we should fork the `branched_llm` dep or patch it to accept a `Task.Supervisor` option.

For this ticket, let's take the pragmatic approach:
1. Check if `ChatOrchestrator.do_process/1` is accessible
2. If yes, start a supervised task that calls `do_process` directly
3. If no, wrap `ChatOrchestrator.run/1` in a supervised Task and monitor the inner PID
