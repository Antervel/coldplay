## T0009 — Move synchronous content classifier HTTP call out of LiveView `handle_event`
git branch: `ai-agent/t0009_offload_classifier_from_handle_event`

**Problem:**
Per RULE 33, long-running work inside `handle_event/3` blocks the LiveView process, freezing the UI for that user.

The call chain is:

1. `ChatLive.handle_event("submit_message", ...)` at `lib/cara_web/live/chat_live.ex:128`
2. → `do_send_message/2` (chat_live.ex)
3. → `ChatService.send_message/3` at `lib/cara/education/chat_service.ex:20`
4. → `run_pipeline/3` at `chat_service.ex:39`
5. → `MessagePipeline.run(:on_message, pipeline_data)` 
6. → `Cara.Plugins.SafetyPlugin.on_message/2`
7. → `Cara.AI.Guard.get_classification_and_score/1`
8. → `Cara.ContentClassifier.classify/1` at `lib/cara/content_classifier.ex:69`
9. → HTTP POST to classifier API at `content_classifier.ex:191` (synchronous, ~5000ms timeout)

This synchronous HTTP call blocks the LiveView process for the entire classifier round-trip (typically 500ms-2000ms, up to 5000ms). During this time, the user sees a frozen UI — no typing, no button clicks, no streaming feedback.

**Solution:**

Offload the classifier call to run asynchronously using LiveView's `assign_async` or a supervised Task. The key insight: the safety classification result determines whether the message gets a normal AI response or a "blocked" response. Instead of blocking the LiveView process, we can:

1. Immediately acknowledge the user's message (show it in the UI)
2. Fire the classifier call asynchronously
3. When the classifier result arrives, either proceed with the AI call or show the blocked message

**Implementation approach:**

**Approach A — Use `start_async/3` (simplest, most idiomatic):**

In `ChatLive`:

```elixir
def handle_event("submit_message", %{"chat" => %{"message" => message}}, socket) do
  # Immediately add user message to the branched_chat
  branched_chat = BranchedChat.add_user_message(socket.assigns.branched_chat, message)
  
  # Start async classification
  {:noreply,
   socket
   |> assign_branched_chat(branched_chat)
   |> start_async(:classify_message, fn -> 
     case ContentClassifier.safe?(message) do
       true -> {:safe, message}
       false -> {:blocked, message}
     end
   end)}
end
```

Then handle the async result:

```elixir
def handle_async(:classify_message, {:ok, {:safe, message}}, socket) do
  # Proceed with normal AI response
  do_send_message(message, socket)
end

def handle_async(:classify_message, {:ok, {:blocked, message}}, socket) do
  # Show blocked response
  branched_chat = ChatService.show_blocked_message(socket.assigns.branched_chat, message)
  {:noreply, assign_branched_chat(socket, branched_chat)}
end
```

**Approach B — Restructure `ChatService.send_message` to accept a callback or be split:**

The `MessagePipeline` runs all plugins synchronously in sequence. If the classification step (inside `SafetyPlugin.on_message`) is the slow part, we should split it out so the pipeline can be paused and resumed after classification.

**Recommended: Approach A** — it's the minimal change, well within LiveView's async pattern, and doesn't require restructuring the entire pipeline.
