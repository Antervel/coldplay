## T0018 — Replace dynamic `String.to_atom` calls with safe alternatives
git branch: `ai-agent/t0018_safe_string_to_atom`

**Problem:**
Per RULE 36, atoms are never garbage collected in the Erlang VM. Any `String.to_atom/1` call using dynamic or user-influenced input is a potential memory exhaustion vulnerability — an attacker could generate enough unique atom values to exhaust the VM's atom table.

Three violations exist:

**Violation 1 — `lib/cara/ai/chat.ex:128`:**
```elixir
defp resolve_model(model) when is_binary(model) do
  case String.split(model, ":", parts: 2) do
    [provider, id] ->
      ReqLLM.model!(%{provider: String.to_atom(provider), id: id})
```

The `provider` comes from the model string, which is either:
- The default model from `Application.get_env(:cara, :ai_model, "openai:cara-cpu")` (config-derived)
- A user selection from the settings page

While the attack surface is low (it's a config value, not direct user input), it's still a dynamic atom from a string that could theoretically be manipulated via `Application.put_env` if an attacker gains config access.

**Violation 2 — `lib/cara_web/markdown_helpers.ex:85,94`:**
```elixir
# Line 85:
{String.to_atom("#{key}_#{suffix}"), fun}

# Line 94:
String.to_atom("#{key}_#{suffix}")
```

The `suffix` is `"mermaid"` or `"katex"` (the `rename_steps/2` function is called with these fixed values from lines 31 and 44). While this is currently safe because the suffix is hardcoded, the function signature `defp rename_steps(doc, suffix)` accepts any suffix — a future change could pass user-controlled data.

**Solution:**

**Fix 1 (`chat.ex`):** Replace `String.to_atom` with `String.to_existing_atom/1` and provide a fallback for unknown providers:

```elixir
defp resolve_model(model) when is_binary(model) do
  case String.split(model, ":", parts: 2) do
    [provider, id] ->
      provider_atom = String.to_existing_atom(provider)
      ReqLLM.model!(%{provider: provider_atom, id: id})
    _ ->
      model
  end
rescue
  ArgumentError ->
    Logger.warning("Unknown provider: #{provider}, using default")
    default_model()
end
```

Wait — `String.to_existing_atom/1` will raise `ArgumentError` if the atom doesn't exist yet. The known providers (`:openai`, `:ollama`, `:anthropic`, etc.) are typically already loaded by `ReqLLM`. But this could break if a new provider is introduced at runtime.

Better approach: Use a mapping from known provider strings to atoms:

```elixir
@known_providers %{
  "openai" => :openai,
  "ollama" => :ollama,
  "anthropic" => :anthropic,
  "google" => :google,
  "mistral" => :mistral,
  "deepseek" => :deepseek,
  "groq" => :groq,
  "together" => :together,
  "fireworks" => :fireworks
}

defp resolve_model(model) when is_binary(model) do
  case String.split(model, ":", parts: 2) do
    [provider, id] ->
      case Map.get(@known_providers, provider) do
        nil -> model  # unknown provider, pass through
        provider_atom -> ReqLLM.model!(%{provider: provider_atom, id: id})
      end
    _ ->
      model
  end
end
```

**Fix 2 (`markdown_helpers.ex`):** The `rename_steps/2` function currently only receives hardcoded suffixes (`"mermaid"` and `"katex"`). To make it safe, either:
- Hardcode the atom names directly (simplest)
- Or validate the suffix against a list of allowed values

Since the atoms being created (`:update_code_blocks_mermaid`, `:inject_init_mermaid`, etc.) are created fresh (not existing), we can't use `to_existing_atom`. The safest fix is to inline the renaming logic:

```elixir
defp rename_steps(doc, "mermaid") do
  # hardcode the mermaid step names
  new_steps = Enum.map(doc.steps, fn
    {:update_code_blocks, fun} -> {:update_code_blocks_mermaid, fun}
    {:inject_init, fun} -> {:inject_init_mermaid, fun}
    {:enable_unsafe, fun} -> {:enable_unsafe_mermaid, fun}
    other -> other
  end)
  # ... similar for current_steps
  %{doc | steps: new_steps, current_steps: ...}
end

defp rename_steps(doc, "katex") do
  # hardcode the katex step names
  ...
end
```
