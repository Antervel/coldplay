## T0008 — Disable or filter OTel log handler to prevent PII export
git branch: `ai-agent/t0008_filter_otel_log_handler`

**Problem:**
Per RULE 30, the current Logger configuration in `config/config.exs:98-107` forwards **every** `Logger.info/error/warning` call to the OpenTelemetry collector via gRPC:

```elixir
config :logger,
  handlers: [
    otel_log_handler: %{
      id: :otel_log_handler,
      module: :otel_log_handler,
      config: %{
        exporter: {:opentelemetry_exporter, %{protocol: :grpc}}
      }
    }
  ]
```

This means even after fixing the log statements in T0007, any future `Logger.info` call that accidentally includes user data would also be forwarded to OTel. The OTel collector persists this data to an observability backend (e.g., Jaeger, Zipkin, Grafana Tempo), creating an additional PII leak vector.

The project also has:
- `config :opentelemetry, :processors, otel_batch_processor: %{exporter: {:opentelemetry_exporter, %{protocol: :grpc}}}` at `config.exs:87-90` — this exports trace spans (which contain span attributes and metadata) to OTel
- `config :opentelemetry_experimental, :metrics_processors, ...` at `config.exs:92-95` — exports metrics

The OTel span processor and metrics processor are fine — they carry structured, non-PII data. The **log handler** is the problem because log messages are freeform text.

**Solution:**

Option A (recommended): Remove the log handler entirely from `config/config.exs`. Move it to `config/prod.exs` only, where an explicit decision to forward logs to OTel can be made with awareness of the PII implications.

Option B: Add a Logger metadata filter that prevents log messages matching certain patterns from being forwarded. However, the `:otel_log_handler` does not natively support filtering by message content.

Option C: Replace the `:otel_log_handler` with a custom handler that scrubs PII before forwarding. This is more work but more robust.

**Recommended approach:** Option A — remove the log handler from `config/config.exs` (which applies to all environments) and only enable it in `config/prod.exs` with a comment warning about PII. Also add a note in `config/runtime.exs` or a README section about the implications.
